# ТЗ

Реализовать справочник терминалов.

## Задача

Создать фоновую службу (BackgroundService) для периодической загрузки справочника терминалов из JSON файла и сохранения данных в PostgreSQL

### Технический стек

- Язык: C# 13 (.NET 9)
- Платформа: ASP.NET Core 9 (Hosted Service, без Web API)
- ORM: Entity Framework Core 9 (PostgreSQL)
- DI: Microsoft.Extensions.DependencyInjection
- Логирование: ILogger (структурированные логи)

## Источник данных

Файл: /files/terminals.json
Расположение: Корневая папка приложения /files/
Формат: JSON-объект с массивом `city`, внутри которого `terminals.terminal`

## Что реализовано

1. Воркер на `BackgroundService`; первый импорт запускается сразу после старта приложения.
2. Простое ежедневное расписание: час + таймзона из конфига.
3. Загрузка справочника из JSON-файла с обработкой ошибок чтения и формата.
4. Маппинг в доменную модель: парсинг адреса, нормализация телефонов, разбор координат, отбрасывание невалидных записей.
5. Подготовка данных перед записью: дедупликация по `Id`, генерация `Uuid` при отсутствии.
6. Синхронизация БД в транзакции по схеме full refresh (delete + insert).
7. Ретраи transient-ошибок БД через `Polly`.
8. Автоприменение миграций при старте.
9. Базовая схема БД и индексы под поиск (`uuid`, `city_code`, `city_code + type`, `address_city`, `address_city + type`).
10. Структурированные логи через `Serilog`.
11. Unit-тесты для парсинга адреса, маппинга и расчёта расписания.
12. Локальный запуск через Docker и `docker-compose`.

## Чего делать не стал

1. Инкрементальный `upsert` не делал; сейчас полная пересборка справочника.
2. Потоковую обработку JSON не делал; файл читается целиком.
3. Чанкования записи в БД сейчас нет.
4. Cron-расписание не подключал, текущего формата “час + таймзона” достаточно.
5. Отдельный слой репозиториев/UoW не добавлял, используется `DbContext` напрямую.
6. Распределённые блокировки для multi-instance сценария не реализованы.
7. Интеграционных тестов с реальной БД пока нет, только unit-тесты.

## Внешние зависимости

### Используются

1. `Polly` для retry-политики на transient-ошибки БД.
2. `Serilog` для структурированных логов.
3. `xUnit` для unit-тестов.
4. Остальное — стандартный стек .NET/EF Core для этой задачи.

### Осознанно не используются

1. `EFCore.BulkExtensions`  
   Не добавлял: текущий объём покрывается стандартной записью через EF Core, а внешняя bulk-библиотека добавляет лишнюю сложность.
2. Cron-библиотеки (`Cronos`, `Quartz.NET`, `Hangfire`)  
   Не добавлял: по ТЗ достаточно одного ежедневного запуска в фиксированный час.
3. Отдельные библиотеки валидации (`FluentValidation`)  
   Не добавлял: для текущего объёма конфигурации достаточно встроенной валидации опций.
4. Отдельные мапперы (`AutoMapper`)  
   Не добавлял: предпочитаю явный маппинг
